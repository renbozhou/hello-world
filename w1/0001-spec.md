# 0001 - Project Alpha：Ticket 标签管理工具（需求 & 设计文档）

## 1. 背景与目标

本项目旨在构建一个**无需用户系统**的轻量 Ticket 管理工具，核心能力是用**标签（Tag）**对 Ticket 进行分类、筛选与管理，并支持按标题搜索。系统采用：

- **数据库**：Postgres
- **后端**：FastAPI
- **前端**：TypeScript + Vite + TailwindCSS + shadcn/ui

目标是提供一个界面清晰、操作顺滑、数据模型简洁但可扩展的单用户工具（“当前用户”即访问者）。

## 2. 范围（Scope）

### 2.1 In Scope（本期必须实现）

- Ticket 的 **创建 / 编辑 / 删除**
- Ticket 的 **完成 / 取消完成**
- Ticket 与标签的 **关联管理**（为 Ticket 添加/移除标签）
- **按标签筛选** Ticket 列表（包含“未打标签”的视图）
- **按标题搜索** Ticket
- 基础的列表与编辑体验（表单校验、错误提示、空状态）

### 2.2 Out of Scope（明确不做）

- 用户注册/登录、权限、多人协作
- 富文本编辑、附件、评论、指派、截止日期、优先级（可作为后续扩展）
- 复杂的全文检索（本期用 ILIKE）
- 实时协作与 WebSocket 推送

## 3. 术语与对象定义

- **Ticket**：一个待办事项/工作项。包含标题与可选描述，具有完成状态。
- **Tag**：标签，用于分类。具有名称（唯一）与可选颜色。
- **TicketTag**：多对多关联表（一个 Ticket 可拥有多个 Tag，一个 Tag 可被多个 Ticket 使用）。

## 4. 典型用户流程

1. 进入首页看到 Ticket 列表（默认显示全部或“未完成优先”）。
2. 点击“新建 Ticket”，输入标题（必填）、描述（可选），保存。
3. 在 Ticket 行或详情中为 Ticket 添加标签（从已有标签选择或新建）。
4. 点击左侧/顶部标签栏选择某个标签，列表只展示带该标签的 Ticket。
5. 在搜索框输入关键词，按标题筛选当前视图的 Ticket。
6. 勾选/点击“完成”切换完成状态；可随时取消完成。
7. 删除 Ticket（需二次确认）。

## 5. 功能需求（Functional Requirements）

### 5.1 Ticket 管理

- **创建 Ticket**
  - 输入：title（必填，1-200 字），description（可选，0-5000 字）
  - 结果：创建成功后在列表中可见，并默认 `is_completed=false`
- **编辑 Ticket**
  - 可编辑字段：title、description
  - 需支持在列表或详情页进入编辑
- **删除 Ticket**
  - 删除后不可恢复（软删除不做）
  - 删除应同时移除其与标签的关联记录
- **完成/取消完成 Ticket**
  - 允许快速切换（toggle）或显式设置
  - 完成时间 `completed_at`：完成时写入，取消完成时清空

### 5.2 标签管理（与 Ticket 绑定的操作为主）

> 需求强调“添加/删除 ticket 的标签”，本期**不强制**提供独立的“标签 CRUD 管理页”，但需要能创建新标签并复用。

- **为 Ticket 添加标签**
  - 支持从已有标签列表中选择
  - 支持在输入框中直接创建新标签（名称唯一，大小写不敏感建议）
- **为 Ticket 移除标签**
  - 从 Ticket 的标签胶囊（badge/chip）上移除
- **按标签查看 Ticket**
  - 选择某个标签后，列表只展示拥有该标签的 Ticket
  - 支持一个特殊筛选：**未打标签**（Ticket 没有关联任何标签）
  - 可选增强：支持“多标签交集筛选”（本期可不做，见 11 扩展）

### 5.3 搜索

- **按 title 搜索 ticket**
  - 搜索行为：在当前筛选（标签/完成状态）基础上再过滤
  - 匹配规则：Postgres `ILIKE '%q%'`，忽略大小写
  - 输入防抖（前端 250-400ms）

### 5.4 列表展示与排序（建议实现）

- 默认排序：`updated_at DESC`（或 `created_at DESC`，二选一，建议 `updated_at`）
- 支持在 UI 上快速切换“全部 / 未完成 / 已完成”（可作为默认的顶栏过滤）
  - 若要极简，也可以先只在后端支持参数，前端后续补

## 6. 非功能性需求（Non-Functional Requirements）

- **易用性**：核心操作 1-2 次点击可完成；空状态清晰。
- **一致性**：API 错误格式统一；前端提示一致。
- **性能**：
  - 列表分页（建议实现，默认 `limit=50`）
  - 标签列表可缓存（前端内存缓存即可）
- **可靠性**：后端请求校验严格；数据库约束保证一致性。
- **可维护性**：清晰的分层（API/Service/Repo），DB migration 可追溯。
- **安全性基础**：无登录但仍需防御常见问题（输入校验、CORS 合理配置）。

## 7. 数据模型设计（Postgres）

### 7.1 表结构（建议）

#### `tickets`

- `id` UUID（PK）
- `title` VARCHAR(200) NOT NULL
- `description` TEXT NULL
- `is_completed` BOOLEAN NOT NULL DEFAULT false
- `completed_at` TIMESTAMPTZ NULL
- `created_at` TIMESTAMPTZ NOT NULL DEFAULT now()
- `updated_at` TIMESTAMPTZ NOT NULL DEFAULT now()

索引建议：

- `idx_tickets_updated_at` on (`updated_at` desc)
- `idx_tickets_is_completed` on (`is_completed`)
- `idx_tickets_title_trgm`（可选，扩展：pg_trgm 加速 ILIKE）

#### `tags`

- `id` UUID（PK）
- `name` VARCHAR(50) NOT NULL UNIQUE
- `color` VARCHAR(24) NULL（例如 `#RRGGBB` 或 Tailwind token 名称）
- `created_at` TIMESTAMPTZ NOT NULL DEFAULT now()

索引建议：

- `uq_tags_name` UNIQUE(`name`)

#### `ticket_tags`

- `ticket_id` UUID NOT NULL (FK -> tickets.id ON DELETE CASCADE)
- `tag_id` UUID NOT NULL (FK -> tags.id ON DELETE CASCADE)
- `created_at` TIMESTAMPTZ NOT NULL DEFAULT now()
- PRIMARY KEY (`ticket_id`, `tag_id`)

索引建议：

- `idx_ticket_tags_tag_id` on (`tag_id`)

### 7.2 约束与一致性策略

- Tag 名称唯一：建议后端在写入前做 trim + 统一大小写策略（例如保存原始，但比较用 lower）。
- 完成状态：
  - `is_completed=true` 时必须有 `completed_at`
  - `is_completed=false` 时 `completed_at` 必须为空
  - 可用触发器/约束或由服务层保证（本期服务层保证即可，扩展可加 DB 约束）。

## 8. 后端设计（FastAPI）

### 8.1 架构建议

- **API 层**：路由、请求/响应 schema（Pydantic）、状态码
- **Service 层**：业务规则（完成状态、标签去重、错误语义）
- **Repo/DB 层**：数据库访问（SQLAlchemy/SQLModel/asyncpg 任一；建议 async + SQLModel 或 SQLAlchemy 2.0）
- **Migration**：Alembic

### 8.2 API 约定

#### 通用

- Base path：`/api`
- Content-Type：JSON
- 时间字段：ISO8601（UTC，带时区）
- 错误响应统一：
  - `{"error": {"code": "...", "message": "...", "details": {...}}}`

#### Ticket

- `GET /api/tickets`
  - Query：
    - `q`：标题搜索（可选）
    - `tag_id`：按标签过滤（可选）
    - `untagged=true`：未打标签过滤（可选，与 `tag_id` 互斥）
    - `completed`：`true/false`（可选）
    - `limit`：默认 50
    - `offset`：默认 0
    - `sort`：`updated_at_desc`（默认），可扩展
  - Response：列表 + 总数（建议）
    - `{ "items": [...], "total": 123 }`
- `POST /api/tickets`
  - Body：`{ "title": "...", "description": "...", "tag_ids": ["..."]? }`
  - Response：创建后的 Ticket（含 tags）
- `GET /api/tickets/{id}`
  - Response：Ticket 详情（含 tags）
- `PATCH /api/tickets/{id}`
  - Body：可部分更新：`title? description?`
- `DELETE /api/tickets/{id}`
  - 204 No Content
- `POST /api/tickets/{id}/complete`
  - 设置完成（也可用 `PATCH` 布尔值；此处更清晰）
- `POST /api/tickets/{id}/uncomplete`
  - 取消完成
- `POST /api/tickets/{id}/tags`
  - Body：`{ "tag_id": "..." }` 或 `{ "tag_name": "..." }`（二选一）
  - 行为：若 `tag_name` 不存在则创建；然后绑定
  - 幂等：重复绑定返回 200 且不新增重复记录
- `DELETE /api/tickets/{id}/tags/{tag_id}`
  - 204 No Content（不存在也可 204 或 404，建议 204 幂等）

#### Tag

- `GET /api/tags`
  - Response：`[{id,name,color,created_at,usage_count?}]`
  - 可选返回 `usage_count`（用于 UI 排序/展示）
- `POST /api/tags`
  - Body：`{ "name": "...", "color": "#..."? }`
  - 若 name 冲突返回 409

### 8.3 响应模型（建议字段）

为便于前端直接渲染，建议 Ticket 与 Tag 返回如下结构（字段可按实现微调，但需保持一致性）。

#### TagDto

- `id`: string(UUID)
- `name`: string
- `color`: string|null
- `created_at`: string(ISO8601)
- `usage_count`: number（可选，仅在 tags 列表返回时提供）

#### TicketDto

- `id`: string(UUID)
- `title`: string
- `description`: string|null
- `is_completed`: boolean
- `completed_at`: string|null
- `created_at`: string
- `updated_at`: string
- `tags`: TagDto[]

#### 列表响应

- `GET /api/tickets` 返回：
  - `items`: TicketDto[]
  - `total`: number
  - `limit`: number
  - `offset`: number

### 8.4 业务规则细节

- **创建/绑定标签的去重**：
  - `tag_name` 写入前执行 trim
  - 冲突策略：若名称已存在则复用已有 Tag（或返回 409；二选一）
  - 本项目更建议“复用”：用户体验更顺滑
- **完成状态切换**：
  - complete：`is_completed=true` 且 `completed_at=now()`
  - uncomplete：`is_completed=false` 且 `completed_at=null`
- **未打标签筛选**：
  - `untagged=true` 表示不存在任何 `ticket_tags` 关联
  - 与 `tag_id` 互斥：同时传入时返回 400

### 8.5 分页策略

- 使用 `limit/offset`（简单直观）
- 默认 `limit=50`，最大 `limit=200`（超出返回 400 或自动 clamp）

### 8.6 错误码建议

- `VALIDATION_ERROR`：请求参数/字段校验失败（422）
- `NOT_FOUND`：资源不存在（404）
- `CONFLICT`：唯一约束冲突（409）
- `INTERNAL_ERROR`：未预期异常（500）

## 9. 前端设计（TypeScript/Vite/Tailwind/shadcn）

### 9.1 页面与路由（建议）

> 若希望最小化路由，也可全部做成单页，但仍建议把筛选条件放在 URL 上，做到可分享/可刷新。

- `/`：Ticket 列表页（含筛选、搜索、创建入口）
- 可选：`/tickets/:id`：Ticket 详情（也可用 Drawer/Dialog，不做独立路由）

URL Query 建议：

- `tag=<tagId|untagged|all>`
- `q=<keyword>`
- `completed=<all|true|false>`

### 9.2 关键 UI 区块

- **顶部工具栏**
  - 搜索框（title）
  - “新建 Ticket”按钮
  - 完成状态过滤（全部/未完成/已完成）
- **标签栏（侧边或顶部）**
  - “全部”
  - “未打标签”
  - 各 Tag（显示名称，可选显示数量）
- **Ticket 列表**
  - 每行：完成勾选、标题、标签 badges、更新时间（可选）
  - 行操作：编辑、删除（下拉菜单）
- **Ticket 创建/编辑**
  - 推荐使用 Dialog 或 Drawer
  - 字段：title、description
  - 标签选择器：多选 + 新建

### 9.3 标签选择器交互（建议）

- 输入框支持：
  - 输入关键字过滤已有标签
  - 回车创建新标签（若不存在）
- 选中标签显示为 badge，支持点击移除
- 若创建标签失败（例如冲突策略选择 409），则提示并引导选择已有标签

### 9.4 状态管理与数据获取

- **数据源**：后端 REST API
- **请求封装**：统一的 `apiClient`（fetch/axios 皆可）
- **缓存/同步**：建议使用 TanStack Query（React Query）以简化列表刷新、失效与乐观更新
- **空/错状态**：
  - 列表为空：提示“暂无 Ticket，创建一个吧”
  - 请求失败：显示错误提示与重试按钮

### 9.5 视觉与组件库约束

- 使用 Tailwind + shadcn/ui 的基础组件：Button、Input、Dialog、Badge、DropdownMenu、Tabs（或自建分段选择器）
- 风格目标：简洁、密度适中、层级清晰

## 10. 关键边界与校验

### 10.1 字段校验

- `title`：
  - 前端：必填、长度限制、去除首尾空格
  - 后端：同样校验，防止绕过
- `description`：允许为空；长度上限
- `tag.name`：1-50 字（或更合理上限），trim

### 10.2 并发与一致性

- 同时修改标签与 Ticket 基本信息的冲突：本期可采用“最后写入覆盖”
- 删除 Ticket 时必须级联清理 `ticket_tags`

### 10.3 幂等性

- 绑定标签接口建议幂等：重复绑定不报错
- 移除标签接口建议幂等：不存在也返回 204

## 11. 测试与验收标准（Acceptance Criteria）

### 11.1 验收用例（必须通过）

- **创建 Ticket**：输入标题后保存，列表立即出现；刷新页面仍存在
- **编辑 Ticket**：修改标题/描述后保存，列表显示更新
- **完成/取消完成**：切换状态后列表视觉变化正确；刷新仍保持
- **删除 Ticket**：二次确认后删除；刷新仍不存在
- **标签添加/移除**：给 Ticket 添加多个标签并移除其中一个，结果正确
- **按标签筛选**：选择某 Tag 后只显示包含它的 Ticket
- **未打标签筛选**：只显示没有任何标签的 Ticket
- **搜索**：输入关键字，仅匹配标题包含关键字的 Ticket（忽略大小写）

### 11.2 测试建议

- 后端：
  - service 层单元测试（完成状态、标签去重、untagged 查询）
  - API 集成测试（CRUD + filter + search）
- 前端：
  - 关键组件的渲染与交互测试（可选）
  - E2E（可选）：创建/筛选/搜索的主流程
